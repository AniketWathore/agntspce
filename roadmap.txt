1. Proxy — local HTTP server that intercepts agent API calls
Goal: Claude Code / Codex sends its API request to localhost → your proxy logs/compresses/routes it → forwards to real Anthropic/OpenAI.
Files to create:
core/proxy/
├── server.ts          # Express/Fastify server, listens on localhost:11435
├── compress.ts        # Compresses large tool-result blocks (Read/Grep/Glob output)
├── trim-prompt.ts     # Trims filler from the newest user message
├── route-model.ts     # Heuristic scorer → swaps model field (haiku↔sonnet↔opus)
└── telemetry.ts       # Logs before/after token counts to SQLite (better-sqlite3)
Key packages: express, better-sqlite3, uuid
How it works:
- Start server on localhost:11435
- Agent sets ANTHROPIC_BASE_URL=http://localhost:11435 or OPENAI_BASE_URL=http://localhost:11435
- Proxy receives request → calls compress() on large blocks → calls trim-prompt() → calls route-model() → logs to SQLite → forwards to real API
- Response goes back to agent unchanged
The HTTP interception already exists in proxy/proxy.js — you can refactor that into this structure.
2. rtk/tokf integration — hook into agent shell commands
Goal: Before a command like git status runs, rewrite it to rtk git status so the output is 90% smaller.
Files to create:
core/hooks/
├── installer.ts       # Runs `rtk init -g --agent <name>` or `tokf hook install --tool <name> --global`
├── gain-parser.ts     # Parses `rtk gain --json` / `tokf gain --json` output for the dashboard
└── index.ts           # Detects agent type and installs the right hook
How it works:
- The user selects "Claude Code" in the pane header
- App calls installer.ts → runs the appropriate install command
- After that, every bash tool call the agent makes is automatically compacted
- Dashboard reads gain stats for token savings display
3. Local model — Ollama for trivial requests
Goal: Simple requests (list files, read this line) are handled by a local model — never hits the cloud.
Files to create:
core/local/
├── server.ts          # Minimal server receiving requests from the proxy
├── classifier.ts      # Scores request complexity (heuristics: message length, keywords)
├── tool-loop.ts       # Minimal tool loop: read_file, write_file, list_dir via Ollama API
└── ollama.ts          # Wrapper around Ollama's localhost:11434 API
How it works:
- Proxy receives request → calls classifier.ts → if score < threshold, sends to local Ollama instead of cloud API
- Ollama runs a small model (Qwen2.5-1.5B) with constrained JSON output
- Response is formatted identically to the cloud API so the agent doesn't know the difference
4. Wire into Electron app — spawn agents in terminal panes
Goal: Each terminal pane actually spawns a real agent CLI process (Claude Code / Codex / OpenCode) with the proxy's env vars.
Changes needed:
electron/
├── main.ts            # Add: spawn agent process via node-pty in the Electron main process
└── preload.ts         # Expose: startAgent(), stopAgent(), onTerminalData()

src/
└── components/
    └── TerminalPane.tsx  # Connect xterm.js to the actual PTY process
Key packages: node-pty (or @electron-remote/pty)
How it works:
- User clicks "Add Panel" → Electron spawns a PTY with: claude command, CWD = workspace path, env = ANTHROPIC_BASE_URL=http://localhost:11435
- xterm.js in the pane connects to the PTY's stdin/stdout
- Input bar at the bottom sends keystrokes into the PTY
- The agent runs normally, but all its API calls go through your proxy
5. Dashboard — show savings and stats
Goal: Real-time view of token savings, model routing, local vs. cloud split.
Files to create:
src/components/
├── Dashboard.tsx       # Stats header: tokens saved, requests handled locally, $ saved
├── TelemetryChart.tsx  # Optional: simple chart showing savings over time
└── RtkGainPanel.tsx    # Shows rtk/tokf gain stats
How it works:
- telemetry.ts (from Step 1) writes to SQLite
- gain-parser.ts (from Step 2) parses rtk/tokf JSON
- Dashboard polls both sources every few seconds and updates the UI
Recommended order to build
Step	Why first
1. Proxy	Everything depends on it — the interception layer
4. Wire into Electron	You need to see it working end-to-end
2. rtk/tokf	Biggest savings, simplest to add
3. Local model	Independent, can be done in parallel
5. Dashboard	Needs data from steps 1-3
